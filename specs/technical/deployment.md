# Deployment

Azure Static Web Apps configuration, CI/CD pipeline, and branch strategy.

---

## Branch strategy

| Branch | Environment |
|--------|-------------|
| `main` | Production  |

Only `main` is deployed. No staging environment. No PR preview environments.

## Azure Static Web Apps deployment

### Create the resource

1. In the Azure portal, create a new **Static Web App** resource.
2. In **Deployment details**, set the source to **GitHub** and select the repository.
3. In **Deployment configuration**, select **Deployment token**.
4. Create the resource.

### Pull the generated workflow

Once the resource is deployed, Azure adds a workflow file under `.github/workflows/`. Pull the repository to retrieve it.

### Update the workflow file

Apply the following changes to the generated file:

- **Remove PR preview support**: remove the `pull_request` trigger and the `close_pull_request_job` job (these contradict the branch strategy).
- **Versions**: verify that `actions/checkout`, `actions/setup-node`, and `Azure/static-web-apps-deploy` use the latest stable versions (`@v4`, `@v4`, `@v1`).
- **No submodules**: ensure the checkout step includes `submodules: false`.

The resulting file should look like:

```yaml
name: Azure Static Web Apps CI/CD

on:
  push:
    branches:
      - main

jobs:
  build_and_deploy_job:
    runs-on: ubuntu-latest
    name: Build and Deploy Job
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: false
          lfs: false

      - uses: actions/setup-node@v4
        with:
          node-version: '22.x'

      - name: Install dependencies
        working-directory: website
        run: npm ci

      - name: Build
        working-directory: website
        run: npm run build

      - name: Build And Deploy
        id: builddeploy
        uses: Azure/static-web-apps-deploy@v1
        with:
          azure_static_web_apps_api_token: ${{ secrets.AZURE_STATIC_WEB_APPS_API_TOKEN_<generated-suffix> }}
          repo_token: ${{ secrets.GITHUB_TOKEN }}
          action: "upload"
          app_location: "./website/build"
          api_location: ""
          output_location: ""
          skip_app_build: true
```

The secret name (e.g. `AZURE_STATIC_WEB_APPS_API_TOKEN_NICE_GROUND_08A335503`) is auto-generated by Azure and added to the repository automatically — no manual secret setup is required.

`skip_app_build: true` tells the Azure SWA action to skip its own Oryx build and upload the already-built `website/build/` directory directly.

## staticwebapp.config.json

Place this file at `website/static/staticwebapp.config.json`. Docusaurus copies every file in `website/static/` verbatim into `website/build/`, so it will be present at the root of the deployed output.

```json
{
  "routes": [
    {
      "route": "/",
      "redirect": "/docs/introduction/",
      "statusCode": 302
    }
  ],
  "responseOverrides": {
    "404": {
      "rewrite": "/404.html",
      "statusCode": 404
    }
  }
}
```

**Root redirect** — visitors arriving at `/` are sent directly to the Introduction page.

**404 override** — unmatched paths are rewritten to Docusaurus's generated 404 page instead of Azure SWA's default. The status code is preserved at 404.

No SPA navigation fallback is needed: Docusaurus generates a full `.html` file for every page at build time.

## Submodule in CI

The submodule (`shared/ProAbonoLive/`) is **not** checked out in CI. The `actions/checkout@v4` step uses a default shallow clone with no `submodules: true` flag.

API reference files are committed to `website/docs/api-reference/` by the local generation command and must be committed before pushing to `main`. See the [pipeline specifications](../pipeline/index.md) for the CI constraint and its implications.
