# Internationalization

Multilingual configuration for the ProAbono API Live documentation website.

## Supported locales

| Locale | Label | URL prefix |
|--------|-------|------------|
| `en` | English | `/en/` |
| `fr` | Français | `/fr/` |

English is the default locale and the authoring language for all source files.

## URL structure

All locales are prefixed — there is no unprefixed root:

```
/en/docs/getting-started/    ← English
/fr/docs/getting-started/    ← French
```

The root path `/` redirects to `/en/` by default. The redirect rule lives in the Azure SWA config (see deployment.md).

## Docusaurus configuration

```js
// docusaurus.config.js
i18n: {
  defaultLocale: 'en',
  locales: ['en', 'fr'],
  localeConfigs: {
    en: { label: 'English', path: 'en' },
    fr: { label: 'Français', path: 'fr' },
  },
},
```

Setting `path: 'en'` on the default locale forces the `/en/` prefix on English URLs. A locale switcher dropdown is enabled via the `localeDropdown` navbar item in `docusaurus.config.js`.

## Filesystem layout

```
website/
  docs/                                            ← English source files (primary locale)
  i18n/
    fr/
      docusaurus-plugin-content-docs/
        current/                                   ← French document translations
          api-reference/
            customer/
              index.mdx
              get.mdx
              ...
          getting-started/
          authentication/
          concepts/
          guides/
      docusaurus-theme-classic/
        navbar.json                                ← translated UI strings (nav items, etc.)
        footer.json
```

French document files mirror the structure of `website/docs/` exactly. A file present in `i18n/fr/` overrides its English counterpart for the `fr` locale; if a file is absent, Docusaurus falls back to the English version.

## What is translated

### Translated
- All prose in hand-authored sections: Getting Started, Authentication, Concepts, Guides
- All prose in extracted API reference pages: resource descriptions, parameter descriptions, response descriptions, notes
- Navigation labels, sidebar labels, page titles
- UI strings: navbar items, footer links, breadcrumb labels

### Always English — never translated
| Element | Examples |
|---------|---------|
| API field names | `ReferenceCustomer`, `IdSegment`, `Currency` |
| Enum values | `Enabled`, `Suspended` |
| HTTP methods | `GET`, `POST` |
| Endpoint paths | `/v1/Customer`, `/v1/Subscriptions` |
| JSON code blocks | All `## Sample` sections |
| Action identifiers | `customer.get`, `subscription.create` |

These identifiers appear verbatim in code and must match exactly what developers send to the API — translating them would be misleading.

## Translation workflow

Translations are performed automatically by Claude. The following section documents the triggers and rules; no manual translation is needed.

### Triggers

Claude translates (or re-translates) a file immediately, in the same task, after any of the following events:

- A file in `website/docs/` is created or modified (hand-authored section)
- `scripts/extract.mjs` is run (API reference regenerated from specs)
- The `shared/ProAbonoLive` submodule is updated (OpenAPI spec changed)
- The user asks to update the site according to the OpenAPI specs

### Term mapping

The file `scripts/locales/fr.json` defines custom term mappings (e.g. `"plan" → "offre"`), tone guidance, and preserve rules for the French locale. Claude reads this file before translating.

### Hand-authored sections

1. Write the English source in `website/docs/<section>/`.
2. Claude creates or updates the French translation at the same relative path under `website/i18n/fr/docusaurus-plugin-content-docs/current/<section>/`.
3. Structure is identical; only prose is translated.

### API reference (generated files)

`docusaurus-plugin-openapi-docs` generates MDX and JSON data files into `website/docs/api-reference/`. The French counterparts are maintained in `website/i18n/fr/docusaurus-plugin-content-docs/current/api-reference/`:

- **MDX files** (`.api.mdx`, `.tag.mdx`, `.info.mdx`): frontmatter values and prose are translated; the binary `api:` field, component imports, JSX structure, and `require()` paths are kept verbatim.
- **JSON data files** (`*.ParamsDetails.json`, `*.StatusCodes.json`, `*.RequestSchema.json`): `description` string values in parameters, responses, and schema properties are translated; all other fields (`name`, `type`, `format`, `enum`, etc.) are kept verbatim.

When the English source is regenerated (spec change), Claude re-translates the affected files.

## Development server limitation

`npm start` (the Docusaurus dev server) serves **only one locale at a time**. By default it serves the default locale (`en`). If you switch to another locale via the navbar dropdown, the browser redirects to `/fr/docs/...` — but that path is not served, producing a 404.

To start the dev server for a specific locale:

```bash
cd website
npx docusaurus start --locale fr
```

Avoid `npm start -- --locale fr`: npm v9+ intercepts `--locale` as an npm config key, stripping it and passing `fr` as the site directory argument, which causes an ENOENT error.

To browse both locales simultaneously, open two terminals:

```bash
# Terminal 1 — English
npx docusaurus start --locale en --port 3000

# Terminal 2 — French
npx docusaurus start --locale fr --port 3001
```

The locale switcher will not navigate between the two ports in this setup, but each locale is independently browsable.

## Locale-specific UI strings

Run the following command to generate the JSON files that Docusaurus uses for UI string translation:

```bash
cd website
npm run write-translations -- --locale fr
```

This creates skeleton JSON files under `website/i18n/fr/docusaurus-theme-classic/`. Fill in the translated values before publishing the French locale.
